<objective>
Make `-CC` (DCS-wrapped) mode work end-to-end. The transport must strip the leading `\033P1000p` introducer from the read stream and emit the `\033\\` terminator on close. Default `-C` mode is untouched.
</objective>

<context>
**SPEC §2.1, §12** — `-CC` mode wraps the protocol stream in a DCS sequence. tmux writes `\033P1000p` (7 bytes: ESC P 1 0 0 0 p) before any protocol output, and `\033\\` (2 bytes: ESC backslash) after the final `%exit`. Without stripping/emitting these, the parser sees garbage on the read side and tmux waits for a clean DCS frame on the write side.

**Current state:** `SpawnOptions.controlControl` already plumbed through `buildArgv`. `-CC` is launched correctly. The DCS framing itself is missing.

**Edge case to handle:** the 7-byte introducer can arrive fragmented (e.g., chunks `\033P10` then `00p%begin...`). The strip logic must buffer until at least 7 bytes are seen, then validate-and-strip, then forward all subsequent data unchanged. After the introducer is stripped, the strip logic is a no-op for the rest of the stream.
</context>

<task type="auto">
  <name>Task 1: Implement DCS framing in spawn.ts</name>
  <files>src/transport/spawn.ts</files>
  <action>
1. Define a constant `DCS_INTRODUCER = "\u001bP1000p"` (7 chars) and `DCS_TERMINATOR = "\u001b\\"` (2 chars). Tag with `// [LAW:one-source-of-truth] DCS frame bytes live here only (SPEC §12).`

2. In `-CC` mode, intercept `child.stdout` data before forwarding to `dataCallbacks`. Maintain two pieces of local state:
   - `introBuffer: string` — accumulator for the bytes seen before the introducer is consumed
   - `introducerStripped: boolean` — flips to `true` after the introducer is recognized and stripped

   On every chunk:
   - **If `introducerStripped` is true:** forward chunk unchanged. (Hot path after first frame.)
   - **If `introducerStripped` is false:** append chunk to `introBuffer`. If `introBuffer.length >= DCS_INTRODUCER.length`:
     - Verify the prefix matches `DCS_INTRODUCER`. If not → emit a transport error via `closeCallbacks` with reason `"invalid DCS introducer in -CC mode"` and stop forwarding data. (Don't crash; surface the error like other transport faults.)
     - If it matches: forward the slice after the introducer to `dataCallbacks`, set `introducerStripped = true`, clear `introBuffer`.
   - Otherwise: keep buffering.

   In `-C` mode (default), bypass all of this — just forward the chunk to `dataCallbacks` exactly as today. The data path for `-C` mode must be byte-identical to its current behavior.

3. In `-CC` mode, change `close()` to:
   - If not already closed and stdin is writable: `child.stdin!.write(DCS_TERMINATOR)`.
   - Then `child.kill()` as before.
   - In `-C` mode, `close()` is unchanged (just `child.kill()`).

4. Use `// [LAW:dataflow-not-control-flow]` comments where the same forwarding pipeline runs for both modes — the variability is in *what* the pipeline does to the bytes, not whether the pipeline runs.

Constraints:
- The `TmuxTransport` interface MUST NOT change.
- The default `-C` mode path through `spawnTmux` must produce byte-identical behavior to before this task (verified by a unit test asserting that an introducer-shaped chunk in `-C` mode is forwarded unchanged).
